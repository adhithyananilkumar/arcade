## About

Arcade is the next evolution of digital education at Amal Jyothi College of Engineering.

Designed as a unified education ecosystem, Arcade brings together learning, assessment, certification, creator tools, and institutional collaboration into a single modern platform.

Whether it's a faculty member publishing a semester course, an organization delivering professional training, or a learner pursuing new skills, Arcade provides the complete infrastructure to create, distribute, and experience education without limitations.

More than an online course platform, Arcade is being engineered as an academic infrastructure that empowers institutions, educators, creators, and learners through technology.

Built with scalability, security, and openness at its core, the platform is designed to evolve continuously—introducing intelligent learning experiences, secure examinations, verifiable digital credentials, and collaborative educational ecosystems for the future.

> "Education should not simply be delivered. It should be experienced, trusted, and continuously evolving."

## Getting Started

To run the Arcade UI locally, follow these steps:

### Prerequisites
- Node.js 18.x or later
- npm, yarn, or pnpm

### Installation

1. Clone the repository:
   ```bash
   git clone <repository-url>
   cd arcade-ui
   ```

2. Install dependencies:
   ```bash
   npm install
   # or yarn install / pnpm install
   ```

3. Run the development server:
   ```bash
   npm run dev
   # or yarn dev / pnpm dev
   ```

4. Open [http://localhost:3000](http://localhost:3000) with your browser to see the UI.

## Content Editor Engine Handoff Log

### 1. Current Status
Editor engine performance pass complete — steady-state typing cost in the course editor is down **~7x** (1,529 ms → 215 ms of blocked main thread per 63 characters) and the worst single stall is down from **936 ms → 128 ms**, measured on a production build. All editor features verified intact.

### 2. What Was Implemented
- **Bubble-layer gating (2026-07-22, second pass)** — the first pass fixed re-render churn but the editor was still slow on *real* lessons. Cause: the original harness passed no `onSave` and used a 63-character document, so it never exercised the costs that scale with document size.
  - Rebuilt the harness to seed a realistic document (`?size=N` paragraphs) and to run the orchestrator's actual save work (`?save=1`). **The save path turned out to be innocent** — 5 ms for a 49 KB body and 60 KB CRDT state. The earlier plan to redesign Yjs persistence for performance reasons was therefore dropped; it remains relevant only for unbounded payload *growth*, not latency.
  - Bisected on a 120-paragraph document: `bare` (no chrome) **0 ms**, toolbar 626 ms, bubbles 1,770 ms, both 2,299 ms. The cost is the library chrome and it scales with document size — every library button calls `editor.can()`, which dry-runs a command against a copy of the document, on every transaction.
  - Gated the 13 node-specific bubbles behind `isPlainTextContext()` in [RichTextBubbles.tsx](apps/creator/editor/components/RichTextBubbles.tsx): when the caret is in ordinary prose none of them can match, so they are not mounted. Text/Link bubbles, the slash menu and the drag handle stay mounted always.
  - Verified: caret in a `codeBlock` mounts 18 extra DOM nodes (the node bubbles) and leaving it unmounts them; the text bubble still appears on selection and hides when collapsed.
  - Production result on the real course-editor configuration: 340 ms → **215 ms**, worst task 348 ms → **128 ms**. On a 39k-character lesson with saving enabled: 286 ms total, worst 150 ms.
- **Editor Engine Performance Pass (2026-07-22)** — profiled with `/dev-editor-perf` against `next build` + `next start`, not dev mode:
  - *Root cause found:* `saveStatus` lived in `ArcadeEditor` state. Each save cycle pushed three updates (`saving` → `saved` → `idle`), and each re-rendered the whole chrome — 43 toolbar buttons plus 16 bubble menus. A **single keystroke** cost 1,416 ms of blocked main thread across three long tasks (806/318/292 ms), and in `chromeless` mode (what the course editor uses) the footer isn't even rendered.
  - Moved save status into an external store consumed via `useSyncExternalStore` — new [SaveStatusFooter.tsx](apps/creator/editor/components/SaveStatusFooter.tsx). Character count now subscribes via `useEditorState` inside the footer. Status changes re-render one `<div>`.
  - `React.memo` on `ArcadeEditor`, `RichTextToolbar`, `RichTextBubbles`. Dropped the dead `editor` prop from `RichTextBubbles` (it reads context), making its memo unconditional.
  - Code-split the chrome: `RichTextToolbar` and `RichTextBubbles` are now `next/dynamic` with `ssr: false`, plus a new [ToolbarSkeleton.tsx](apps/creator/editor/components/ToolbarSkeleton.tsx). Moved the Excalidraw / easydrawer / KaTeX stylesheets into `RichTextBubbles` so they travel in the deferred chunk instead of the route entry.
  - Title fields (course, lesson, quiz) moved to [DebouncedTitleInput.tsx](apps/creator/components/DebouncedTitleInput.tsx) — keystrokes no longer re-render the 1,500-line orchestrator.
  - Version-history preview now uses `TiptapContentView` (JSON → React) instead of mounting a second full Tiptap instance.
  - Autosave: skips no-op saves via a body fingerprint, defers the `localStorage` mirror to `requestIdleCallback`, and stops awaiting the auto-snapshot POST. `bytesToBase64` no longer spreads 32k-element subarrays.
  - Fixed a latent stale-closure bug: `debouncedSave` captured `onSave` from first render; it now reads through a ref. Was masked only by the editor's `key` remount.
  - `openLesson` now runs the outgoing flush and the incoming document fetch concurrently instead of head-to-tail.
- **Next.js workspace root pinned** — a stray empty `package-lock.json` in the parent directory made Turbopack infer the wrong root, breaking the React Client Manifest for `global-error.js`. `turbopack.root` is now set explicitly in `next.config.ts`.
- **Second Merge of `auth` Branch**: Pulled the latest changes from `auth` and successfully merged them into `studio`. This brought in additional public course pages, fonts, and minor dashboard navigation updates without conflicts.
- **Merge `auth` Branch**: Successfully merged the latest changes from the `auth` branch into `studio` in both frontend and backend repositories, bringing in dashboards, onboarding features, and channel management without disrupting the content editor engine. Resolved merge conflicts in `package.json` for rich text dependencies.
- **Navbar Cleanup**: Removed course description input field and free/paid pricing model dropdown selector from the header of [CourseEditorShell.tsx](file:///c:/Users/athul/OneDrive/Documents/Akash%20A/project_arcade/arcade/features/content/course/components/CourseEditorShell.tsx).
- **Settings Description Editing**: Swapped the static description display with a dynamic, left-aligned `<textarea>` on the settings panel that triggers debounced updates via `onDescriptionChange` back to the parent component and the backend API.
- **Collapsible Sidebar**: Implemented a transition-based collapsible side panel. The panel shrinks to a sleek sidebar rail containing a larger, responsive toggle button (`PanelLeftClose` / `PanelLeftOpen` icons resized to `20px`). Other elements are dynamically hidden and disabled when closed.
- **Last Edited Metric on Dashboard**: Modified [page.tsx](file:///c:/Users/athul/OneDrive/Documents/Akash%20A/project_arcade/arcade/app/%28authenticated%29/studio/page.tsx) to label the course cards' indicator with "Last edited: " and formatted it as a full date-time string (including hour, minute, and AM/PM). *(Note: this dashboard now lives at `/studio`, not `/content` or `/dashboard` — see "Unified Content Model" in `docs/architecture/platform-architecture.md`.)*
- **Merge Conflict Resolution in Rich Text Extensions**: Resolved conflict in [index.ts](file:///c:/Users/athul/OneDrive/Documents/Akash%20A/project_arcade/arcade/features/content/editor/extensions/index.ts) by retaining both the `yjs` collaborative dependency type import (`import type * as Y from "yjs"`) and the `RoadmapNode` extension import (`import { RoadmapNode } from "../roadmap/extensions/roadmap"`).
- **Yjs Duplicate Import Fix (2026-07-22)**: Fixed the `"Yjs was already imported"` console error by adding a `resolveAlias` for `yjs` in the Turbopack configuration inside `next.config.ts`. This forces Next.js to use a single instance of `yjs`, resolving the constructor check errors.
- **Tiptap flushSync Error Suppression**: Suppressed the React 19 `flushSync was called from inside a lifecycle method` console error thrown by `reactjs-tiptap-editor` when mounting nodes. This was achieved by monkey-patching `console.error` locally in `ArcadeEditor.tsx`, silencing the noisy but benign warning without breaking upstream library functionality.
- **R2 CORS Diagnosis**: Diagnosed the `Failed to fetch` error during Cloudflare R2 media uploads. Identified that the backend's automatic `PutBucketCorsRequest` fails with `AccessDenied` because the R2 API token provided in `.env.r2.local` only has Object-level permissions.

### 4. Key Decisions & Rationale
- **Save status lives outside React state**: the autosave indicator is intentionally an external store rather than `useState`. It is the difference between a status change re-rendering one `<div>` and re-rendering 59 chrome components. If you reintroduce `useState` here, you reintroduce the 1.4 s-per-keystroke stall.
- **Extension set was deliberately NOT trimmed**: Tiptap builds its schema from the extension array at construction. Lazily omitting Excalidraw/Mermaid/KaTeX would make documents containing those nodes unparseable. Code-splitting targets the *chrome*, which is safe to defer; the extension array stays complete. `lowlight`'s `common` language set was kept for the same reason — narrowing it silently downgrades highlighting on existing content.
- **Version previews bypass Tiptap**: consistent with the platform rule that read paths never load Tiptap. `TiptapContentView` already existed for the learner renderer and is the correct tool for a read-only snapshot.
- **Yjs persistence contract left unchanged**: the backend stores `ydocState` as an opaque blob and never decodes it, which is what keeps single-writer version history free of a JVM Yjs implementation. Moving to incremental updates would need an append-log table, a compaction policy, and a Flyway migration — flagged as a proposal, not done unilaterally (see Known Issues).
- **Left-aligned Textarea**: Ensuring textarea text is left-aligned to align with typical multi-line reading patterns and input UX expectations, rather than using the right-aligned format designed for static labels.
- **Header Icon Size & Border Removal**: Increased toggle button size to `20px` to give it a strong visual weight on the narrow collapsed rail and conditionally omitted the bottom border when collapsed to keep the rail structure minimal and elegant.

- **Remaining cost is inside the library's buttons.** After gating, what's left is the toolbar's 43 buttons plus the text bubble's own button row, each re-evaluating `editor.can()` per transaction. This is `reactjs-tiptap-editor` internals, not our code. Reducing it further means either trimming the toolbar, or replacing the library chrome with a lean in-house toolbar that subscribes once instead of ~50 times. Not attempted — it would change authoring UX and needs a product decision.
- **Duplicate Yjs instances.** The console logs `"Yjs was already imported. This breaks constructor checks"` — the editor library bundles its own copy alongside ours. Pre-existing, not introduced here, but it makes cross-instance `instanceof` unreliable (which is why `isPlainTextContext` uses a structural check for `NodeSelection`). Worth resolving via a `resolutions`/`overrides` entry for `yjs`.
- **Autosave payload grows without bound.** `createYDoc()` uses `gc: false` (deliberate — it is what makes snapshots restorable), but `Y.encodeStateAsUpdate` sends the *entire* accumulated CRDT history on every save. Measured at 60 KB for a 39k-character lesson, so this is a *storage/bandwidth* concern rather than a latency one. Requires the append-log redesign; **needs architectural approval before starting.**
- **`/dev-editor-perf` still ships in production builds.** It is marked "TEMPORARY — DELETE" in its own header but is genuinely useful for this work, and was extended with a `?mode=chromeless` variant that mirrors the real course-editor configuration. Decide whether to keep it (and gate it) or remove it.

- Ensure you are working on the `studio` branch in both repositories.
- Run `npm install` inside the `arcade/` directory if needed, then run `npm run dev` to start the frontend.
